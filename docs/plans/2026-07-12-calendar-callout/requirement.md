## 背景

项目已支持 Obsidian/GitHub 风格的 callout 语法（`> [!NOTE]` 等，详见 [remark-alert.ts](../../../web/src/utils/remark-plugins/remark-alert.ts)）：在普通 memo 正文中写一段约定语法，前端识别后渲染成特殊卡片，不需要额外的文档类型、不需要表单、不需要跨文档查询。

也已支持独立的 `view` 文档类型（详见 [gallery-view 需求](../2026-07-06-gallery-view/requirement.md)），用于"按 scope 规则跨文档聚合展示"的场景，但代价是要有专属文档类型、结构化 JSON content、预置表单。

本需求要解决的场景不是"跨文档聚合"，而是"日常记流水账时，把当天做的事按日期分组记录，并且比纯文本更好看一点"——这与 callout 是同一量级的问题：只在**单篇文档内部**、**渲染层面**做文章，不引入新文档类型、不做知识库范围的查询聚合。因此选择 callout 的技术路线，而非 view 文档的技术路线。

## 术语

- **calendar 代码块**：本需求新增的约定语法，使用带语言标签的 fenced code block（```` ```calendar ```` ... ```` ``` ````），与项目现有的 mermaid 代码块（[MermaidBlock.tsx](../../../web/src/components/MemoContent/MermaidBlock.tsx)）是同一机制、同一层级的特殊渲染分发。
- **日期分组**：calendar 代码块内部，用一个孤立的列表项 `- YYYY-MM-DD`（不带 checkbox）标记一个日期分组的开始，其后连续的列表项都归属该日期，直到下一个日期分组行或代码块结束。

## 需求概述

1. 用户在任意 memo/文档正文中，写一个语言标签为 `calendar` 的代码块，内部用「日期行 + 若干列表项」的结构记录待办/日志（具体语法见下）。
2. 预览该文档时，前端识别这个代码块，渲染成一个**可按月翻页的月历网格**：
   - 网格部分：标准月视图（周一~周日或周日~周六，对齐当前项目/用户的历史习惯即可），每个日期格子只显示日期数字 + 一个"当天有事项"的提示（圆点或事项数量角标），不在格子内直接排列事项文字。
   - 月份导航：网格上方有"上一月/下一月"按钮（以及可选的"回到今天"），点击后切换网格显示的月份，不影响解析出的数据本身。
   - 详情交互：点击某个日期格子，网格下方出现"当天事项详情区"，展示该日期分组下的事项列表（含 checkbox/纯文本两种条目）；未选中任何日期时，详情区为空或显示提示文案。
3. 编辑态（写作时）不改变现有 markdown 编辑器行为——用户仍然是在写代码块，普通 markdown 编辑体验（含语法高亮，如果编辑器本身对代码块有高亮的话）保持不变，只有*预览渲染*不同。不新增专属编辑表单，不做所见即所得的日历交互控件（月历网格是**预览态**的渲染产物，不是编辑态的输入控件）。
4. 该代码块内容就是普通 markdown 文本的一部分，随文档一起保存，不产生独立的结构化字段、不需要新的 DB 字段或文档类型。

## 语法定义

确定的语法样例：

````
```calendar
- 2026-07-13

- [ ] 制定规范的Vision，使命，年度规划
- [ ] 完成英文句型拼定 和其他雅思学习
- [ ] 推进NYC项目 3小时以上

- 2026-07-08

- [ ] memos vps+host 密钥mac客户端加密
- [ ] Life知识库结构初始化
- [ ] 项目管理 HTML页面
```
````

规则：

- 代码块语言标签固定为 `calendar`（大小写不敏感），复用现有 fenced code block 的语言分发机制（与 `mermaid` 同一处判断逻辑）。
- 块内按行扫描：
  - 形如 `- YYYY-MM-DD`（仅日期，无 checkbox、无其他文本）的列表项，视为**日期分组标题行**，开启一个新分组。日期格式必须是 `YYYY-MM-DD`，不满足格式的行不触发分组、按普通事项处理。
  - 其余列表项（`- [ ] xxx` / `- [x] xxx` / `- xxx`）归入"当前最近一个日期分组"下的事项列表。
  - 分组标题行出现之前的列表项（不属于任何日期），归入一个不显示日期头的"未分组"区块，展示在最前面。
  - 空行仅作为可读性分隔，不影响分组归属。
- 分组内事项支持两种形态：带 checkbox（`- [ ]` / `- [x]`）渲染成可勾选任务项；不带 checkbox 的 `- ` 渲染成纯文本条目（不出现 checkbox）。
- 分组按块内出现顺序渲染（不做日期排序、不做"今天/未来/过去"分类），所写即所得，符合"流水账手记"的使用场景。

## 明确不做的事（对齐 callout 路线，避免范围蔓延）

1. **不做"格子内直接铺陈事项文字"的布局。** 格子只负责"有没有事项/有几条"的提示（圆点/角标），事项详情统一在网格下方的详情区展示，避免格子被撑高、月视图变形，也避免在窄屏下无法阅读。
2. **不做跨文档聚合。** 只渲染当前这一个代码块里手写的内容，不扫描知识库里其他文档的日期/待办信息。跨文档聚合是 `view` 文档类型的问题域，见 [gallery-view 需求](../2026-07-06-gallery-view/requirement.md)，如果未来确有该诉求，应扩展 view 体系而非本机制。
3. **勾选交互不回写文档。** 与现有 markdown 任务列表项（`- [ ]`）在本项目里的行为保持一致——calendar 代码块内的 checkbox 是否支持点击勾选、点击后是否回写 content，跟随现有任务列表项的既有行为，本需求不新增额外的持久化能力。
4. **不做日期分组的自动排序、去重、校验提示。** 用户写错日期格式、日期重复、乱序，都原样按扫描结果渲染，不做"帮用户整理"的智能行为。
5. **不新增编辑器侧的辅助 UI**（如"插入 calendar 代码块"的工具栏按钮、日期选择器辅助输入）。本期只做渲染，编辑体验保持"手写 markdown"。

## 预览态流程

1. 打开包含 `calendar` 代码块的文档，markdown 渲染管线在遇到该代码块时，分发给专属渲染组件（与 mermaid 同一处 `language === "xxx"` 判断逻辑）。
2. 渲染组件解析代码块文本内容为「分组数组」，每组含可选的日期标题 + 事项列表（"未分组"分组不参与网格展示，因为它没有具体日期、放不进任何格子——见下方"未分组事项的处理"）。
3. 网格初始显示月份：取数据中**最晚（最大）**的日期分组所在的月份；若代码块内容为空或解析不出任何带日期的分组，网格默认显示当前自然月。
4. 网格渲染当月每一天，有事项的日期显示提示标记（圆点/数量角标），当月之外、用于对齐星期的格子做留白/弱化处理，不可点击。
5. 用户点击上一月/下一月按钮，网格切换到对应月份重新渲染；切换月份不影响已选中的日期高亮状态（若切走后原选中日期不在当前月，详情区清空）。
6. 用户点击某个有效日期格子：若该日期在解析出的分组里有对应数据，下方详情区展示该分组的事项列表；若该日期没有数据（用户点了空日期），详情区展示"当天无记录"之类的空态文案，而不是报错或不响应。

### 未分组事项的处理

语法定义中允许"日期行之前出现的事项"归入一个无日期的分组。这类事项没有具体日期，无法放进月历格子，因此在网格视图里改为：渲染在网格上方（网格之前）的一个固定小区块，标题类似"未分组"，始终展示、不随月份切换变化。

## 与已有机制的关系

- 技术路线上是 callout（remark-alert / mermaid）机制的同类扩展，不是 view 文档机制的扩展。二者面向的场景不同：callout 类解决"单文档内的语义化片段渲染"，view 类解决"跨文档聚合展示"。
- 不影响、不修改现有 `calendar` 一词以外的其他语言标签的代码块渲染（如 mermaid、其他语言高亮）。

## 后续可迭代方向（不在本次范围内，仅记录以防遗忘）

- 本机制的数据源永远只是"当前这一个代码块"。若用户确有"看全知识库范围内所有日期分组、跨文档聚合到同一个月历上"的诉求，那是另一个独立量级的需求，应做成真正的 `view` calendar 类型（结构化 JSON + 预置表单 + 实时跨文档查询），参照 [gallery-view 需求](../2026-07-06-gallery-view/requirement.md)的技术路线另行设计，不是本机制的简单升级。
- "回到今天"按钮、键盘翻月、格子内圆点按事项数量变色分级等细节，留待渲染组件落地后再迭代，不影响本次的骨架设计。
